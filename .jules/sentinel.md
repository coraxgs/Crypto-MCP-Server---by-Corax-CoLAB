## 2024-03-14 - Removed hardcoded dashboard password and fixed auth timing attack
**Vulnerability:** The application was using a hardcoded `'admin'` default password for `DASHBOARD_PASSWORD`, which leads to an insecure out-of-the-box state. Additionally, `app.use('/api', ...)` used a simple string comparison `token !== DASHBOARD_PASSWORD` which is vulnerable to timing attacks. Finally, backend AI endpoints like `/order/pending` lacked any authentication, allowing anyone on the network to push orders into the system. Order parameters (`amount`, `price`) also lacked boundary validation (could be negative or NaN).
**Learning:** Default passwords are a frequent anti-pattern for "easy setup." Replacing the hardcoded password with an explicitly generated, securely random session password maintains usability for local instances while drastically improving the default security posture.
**Prevention:** Always use `crypto.timingSafeEqual` for secret comparisons. Generate strong session keys at runtime if environment variables are missing, instead of falling back to trivial passwords. Enforce parameter boundaries and type checking on all routes, even internal ones. Always restrict endpoints meant for local system interaction (e.g. AI logic passing data) to `localhost` to mitigate lateral movement on the network.
